## 1. 문서 목적

이 문서는 프로젝트 목표, 공개 전략, 그래픽 콘셉트, 에셋 제작 범위, 전투 검증 기준을 팀이 같은 의미로 이해하도록 만든다. 문서 자체를 완벽하게 만드는 것보다 `공개 → 관찰 → 수정 → 재검증`의 반복 속도를 높이는 것이 목적이다.

표시 규칙은 다음과 같다.

- **확정**: 팀 대화와 현재 구현이 일치하는 기준
- **잠정**: 초기 공개에는 적용하지만 외부 반응에 따라 바꿀 수 있는 기준
- **검증 필요**: 사람 플레이 또는 데이터로 확인해야 하는 가설
- **보류**: 지금 결정해도 구현 가치가 낮거나 정보가 부족한 항목

## 2. 프로젝트 목표와 공개 전략

Coin Combat의 현재 목표는 처음부터 상업용 완성품을 만드는 것이 아니다. 동전을 스킬에 배치하고 앞면·뒷면의 결과를 캐릭터별 기믹으로 활용하는 전투가 실제로 이해 가능하고 재미있는지 검증한다.

진행 순서는 다음과 같다.

```text
핵심 전투 구현
→ 무료 웹 프로토타입 공개
→ 사용자 행동과 피드백 수집
→ 빠른 수정과 반복 검증
→ 게임성과 시각 완성도 향상
→ 필요할 때 정식 플랫폼·엔진 확장 검토
```

현재 단계에서 판매량보다 중요한 결과는 다음과 같다.

- 실제 게임 배포와 운영 경험
- 외부 사용자의 이해 실패와 이탈 지점 발견
- 문제를 근거와 함께 수정한 기록
- 기획·개발·테스트 사이의 결정 과정 축적
- 수정 이후 결과를 다시 확인하는 반복 능력

## 4. 포지셔닝과 타깃

### 한 줄 정의

> 동전을 스킬에 장전하고 앞면과 뒷면의 불확실성을 배치해 캐릭터별 전략을 구축하는 턴제 로그라이트

핵심 타깃은 턴제 전략, 덱빌딩 로그라이트, 확률과 자원 배분, 캐릭터별 빌드에 관심 있는 웹·PC 사용자다. 장르 특성상 20~30대 남성 비중이 높을 가능성은 있지만, 이를 이유로 게임 전체를 특정 성별·취향 전용으로 고정하지 않는다.

다음 두 목표를 동시에 만족시킨다.

1. 캐릭터만 보고도 플레이를 시작할 시각적 동기가 있어야 한다.
2. 캐릭터 표현 때문에 핵심 전투를 시도하기 전 이탈하는 사용자가 과도하게 생겨서는 안 된다.

서브컬처 캐릭터성은 진입 요소로 쓰되, 성적 표현만으로 관심을 끌지 않는다. 캐릭터의 외형은 속성, 무기, 성격, 상태이상, 대표 빌드와 연결되어야 한다.

## 5. 그래픽과 캐릭터 방향

### 선택 화면

- 전신 또는 상반신 스탠딩 일러스트 사용
- 대표 색상, 속성, 무기, 전투 스타일을 첫 시선에 전달
- HP·시작 동전·시작 스킬·고유 특성의 정보 가독성 유지
- 일러스트는 카드 정보와 경쟁하지 않고 캐릭터 선택 동기를 강화

### 전투 화면

- 현재 도트/SD 스프라이트와 실루엣 중심 연출 유지
- 공격, 피격, 방어, 상태이상 피드백이 세부 묘사보다 우선
- 캐릭터별 핵심 색과 무기가 작은 크기에서도 구분되어야 함

### 캐릭터 로스터

초기에는 이미 기획·구현된 캐릭터를 사용한다. 최종 성비는 고정하지 않으며 선택률, 남성 캐릭터 요구, 캐릭터별 유입·이탈 반응을 본 뒤 결정한다. 캐릭터 데이터 구조는 성별과 무관하게 확장 가능해야 한다.

## 6. 에셋 사용과 제작 기준

고품질 캐릭터 원본은 다음 용도로 사용한다.

- 캐릭터 선택 화면
- 기획 문서의 대표 이미지
- 외부 테스트 모집 이미지와 짧은 영상
- 추후 최종 에셋 제작 기준

최종 반영 전 확인 항목:

- 캐릭터 ID와 원본 파일의 명시적 매핑
- 사용 권리와 생성·후보정 기록
- 투명 배경 또는 일관된 마스킹
- 화면 비율과 시선 방향 통일
- 모바일에서 텍스트를 가리지 않는 크롭
- 무기·손·대표 속성의 오류 확인

원본이 없는 캐릭터를 다른 캐릭터 이미지로 조용히 대체하지 않는다. 임시 대체가 필요하면 UI와 문서에 임시 상태를 명시한다.

에셋 우선순위는 `식별 가능성 → 정보 전달 → 대표 색상 → 전투 피드백 → 최종 품질` 순서다.

## 7. 애니메이션 방향

Spine 학습과 리깅은 현재 필수 작업이 아니다. 선택 화면에서 움직임이 필요하면 눈 깜박임, 호흡, 머리카락·옷자락, 무기와 속성 이펙트의 약한 점멸처럼 2~4프레임 또는 레이어 기반 움직임부터 적용한다.

강한 모션은 선택 확정, 피해, 위험, 보상처럼 상태가 바뀌는 순간에 집중하고 `prefers-reduced-motion`을 존중한다.

## 8. 전투 시스템 기준

턴당 스킬 3회 같은 전역 제한은 사용하지 않는다. 이 제한은 버프 이후 공격, 동전 드로우·보존, 동전 재사용처럼 캐릭터가 준비한 연계를 중간에서 끊는다.

행동 강도는 개별 스킬로 조절한다.

- 기본 공격·방어: 쿨다운 0, 반복
- 일반 스킬: 역할과 강도에 맞춘 쿨다운
- 강한 연계 스킬: 긴 쿨다운
- 전투 판도를 크게 바꾸는 스킬: 전투당 1회

기본 방어 반복으로 장기전이 고착될 경우 전역 캡을 다시 넣지 않는다. 적 강화, 방어 관통, 상태이상, 장기전 패시브, 보상 효율, 동전 구성으로 해결한다.

수치와 정확한 규칙은 [`PRD.md`](./PRD.md)와 코드·테스트가 정본이다.

## 9. 캐릭터별 플레이 경험

| 캐릭터 | 핵심 경험 |
|---|---|
| 화염 격투가 | 화상을 누적해 터뜨리거나 과열로 공격을 강화 |
| 수호자 | 방어와 반격, 마나 기반 생존 전술 |
| 번개 결투사 | 감전과 르미즈로 연속 공격과 재사용을 설계 |
| 냉기 도적 | 동전 보존과 선택적 드로우로 다음 턴을 통제 |
| 마도기사 | 소환 장비를 축적·지휘해 전투 규모를 성장 |
| 혈액 마검사 | 자해·흡혈과 혈마검 성장의 위험을 조절 |

외형, 스킬, 동전 효과, 상태이상은 같은 플레이 경험을 설명해야 한다.

## 10. AI 활용 원칙

AI는 아이디어 초안, 누락된 예외 검토, 수치 비교, 반복 시뮬레이션, 문서 정리, 개발 사양 변환에 사용한다. 재미와 캐릭터 경험의 최종 판단은 실제 플레이와 기획 의도를 기준으로 사람이 내린다.

AI가 명시되지 않은 빈칸을 채운 경우 `제안`, `AI 임의 판단`, `플레이테스트 필요`, `수치 검증 필요`, `보류` 중 하나로 표시한다. 자연스럽게 작성됐다는 이유만으로 확정 기획으로 취급하지 않는다.

## 11. 플레이테스트 기준

내부 테스트는 기능·회귀·소프트락·과도한 빌드·의미 없는 선택·UI 문제를 찾는다. 외부 테스트는 정답을 알려주지 않고 다음을 관찰한다.

- 동전을 스킬에 장전하거나 소비해야 한다는 것을 이해했는가
- 앞면·뒷면과 속성 효과를 이해했는가
- 어느 동전을 어느 스킬에 쓸지 고민했는가
- 캐릭터별 차이를 느꼈는가
- 다음 턴을 계획했는가
- 패배 원인을 설명할 수 있는가
- 한 번 더 플레이하고 싶은가

초기 지표는 첫 전투 완료율, 이탈 화면, 캐릭터 선택률, 스킬 사용률, 빌드 분포, 전투 시간, 재도전 의향, 이해 실패 원인이다. 자동 시뮬레이션은 회귀와 밸런스 위험을 찾지만 재미 판정을 대체하지 않는다.

## 11.1 운영·배포 및 분석 확장 계획

초기 외부 플레이테스트에서는 GitHub Pages와 GitHub Actions 배포를 유지한다. 현재 목표는 정적 웹게임의 전투 경험을 빠르게 검증하는 것이므로, Cloudflare Pages 이전은 지금의 필수 작업이 아니다.

Cloudflare는 다음 중 하나가 실제로 필요해질 때 재검토한다.

- 커스텀 도메인, 배포 미리보기 또는 CDN 운영 요구
- 익명 게임 행동 이벤트와 피드백을 받는 Worker 기반 API
- GitHub Pages가 테스트 배포·운영을 방해한다는 확인

분석은 두 단계로 나눈다.

1. **방문 분석**: 방문·유입·기기·실사용 성능은 Cloudflare Web Analytics 도입 여부를 검토한다.
2. **게임 행동 분석**: 캐릭터 선택, 전투 시작·종료, 런 종료처럼 제품 판단에 필요한 최소 이정표만 익명 집계한다. 매 행동·개인 식별 정보·원본 플레이 로그는 외부로 보내지 않는다.

현재 UI의 사람 플레이 텔레메트리는 로컬 JSON 내보내기만 제공한다. 외부 이벤트 수집은 테스트 참여자가 충분히 모이고 수집 항목·보관 기간·고지 방식을 합의한 뒤 Cloudflare Worker 등 별도 API로 구현한다.

## 12. 우선순위

### P0 — 공개 가능한 핵심 게임

오류 없이 캐릭터 선택부터 한 런의 승패까지 진행할 수 있어야 한다.

### P1 — 플레이 경험 검증

튜토리얼, 툴팁, 전투 로그, 피드백 설문, 행동 기록, 밸런스 수정에 집중한다.

### P2 — 시각·청각 완성도

선택 일러스트, 전투 스프라이트, 속성 이펙트, UI, 사운드, 홍보 영상의 품질을 높인다.

### P3 — 확장과 정식화

엔진 포팅, 스토어 출시, 추가 막·캐릭터·메타 성장·유료화는 검증 결과에 따라 결정한다.

## 13. 보류 항목

- 최종 게임명과 저장소 개명 여부
- 최종 캐릭터 성비와 공개 버전 캐릭터 수
- Unity 등 별도 엔진 포팅 여부와 시점
- Cloudflare Pages 이전 여부와 시점
- Cloudflare Web Analytics 및 익명 게임 행동 이벤트 수집 도입 여부
- 스토어 출시 및 유료 판매·과금 여부
- Spine 도입
- 최종 일러스트 제작 방식
- 최종 런 길이와 장기 성장 구조
- 기본 공격·방어에 추가 제한이 필요한지

보류 항목은 문서 완성을 위해 임의로 확정하지 않는다.

## 15. 합의문

Coin Combat는 캐릭터의 시각적 매력을 진입 요소로 활용하지만 캐릭터 디자인만으로 승부하지 않는다. 선택 화면은 고품질 일러스트, 전투 화면은 가독성 높은 도트/SD 스프라이트로 역할을 분리한다.

프로젝트는 무료 웹게임으로 먼저 공개하고, 실제 사용자의 이해 실패와 재미없는 지점을 찾아 빠르게 개선한다. 전투는 전역 행동 캡 대신 스킬별 쿨다운과 전투당 1회 규칙으로 조절한다.

AI는 초안과 검증을 돕지만 최종 판단은 사람이 직접 플레이하고 내린다. 문서 작성이 진행을 막지 않도록 최소 정본과 의사결정 기록을 유지하며, 세상에 공개한 결과를 다음 수정의 근거로 사용한다.
