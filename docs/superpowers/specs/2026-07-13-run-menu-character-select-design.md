# 런 메뉴 및 캐릭터 선택 개선 설계

## 1. 목적

플레이어가 진행 중인 런에 갇히지 않고 여러 캐릭터와 빌드를 반복 테스트할 수 있게 한다. 게임 접속 시 명시적인 타이틀 화면을 제공하고, 런 도중에도 저장 불러오기·새 런·타이틀 복귀를 선택할 수 있게 한다. 캐릭터 선택 화면에는 이미 제작된 전투 스프라이트를 함께 노출해 이름만으로 고르는 문제를 해결한다.

## 2. 확정된 사용자 흐름

### 일반 접속

1. 쿼리 없는 주소로 접속하면 타이틀 화면을 표시한다.
2. 유효한 저장이 있으면 `이어하기`를 활성화한다.
3. 저장이 없으면 `이어하기`를 비활성화하고 `새 런 시작`을 주 행동으로 표시한다.
4. `새 런 시작`은 캐릭터 선택 화면으로 이동한다.
5. 캐릭터를 선택하면 새 시드와 해당 캐릭터의 시작 구성으로 런을 시작한다.

### 런 도중

모든 전투·보상·상점·이벤트·휴식·보물·노드 선택 화면에서 `메뉴`를 열 수 있다.

- `계속하기`: 메뉴를 닫고 현재 화면으로 돌아간다.
- `마지막 저장 불러오기`: 저장된 런 상태를 다시 읽는다. 전투 중 사용한 동전과 행동은 저장 대상이 아니므로 현재 전투를 처음부터 다시 시작할 수 있다는 확인을 먼저 표시한다.
- `새 런 시작`: 현재 저장을 삭제한다는 확인 후 캐릭터 선택 화면으로 이동한다.
- `타이틀로 나가기`: 현재 런 저장을 유지하고 타이틀 화면으로 이동한다.

웹 브라우저 탭을 닫는 기능은 제공하지 않는다. `나가기`는 항상 타이틀 복귀를 의미한다.

## 3. 저장 및 부팅 계약

- 자동 저장 규칙은 유지한다.
- 쿼리 없는 일반 접속은 저장을 자동 재개하지 않고 타이틀을 먼저 표시한다.
- 타이틀의 `이어하기`가 기존 자동 재개 로직을 명시적으로 실행한다.
- `?seed=`, `?encounter=`, `?skills=`, `?character=` 테스트 부팅 경로는 기존처럼 타이틀을 건너뛴다.
- `?select=1`은 캐릭터 선택 화면으로 직접 진입한다.
- 손상되거나 지원하지 않는 저장은 기존 복구 화면을 유지한다.
- 저장소 접근이 불가능하면 `이어하기`를 비활성화하되 새 런은 메모리 세션으로 시작할 수 있다.
- 단일 저장 슬롯만 사용한다. 다중 슬롯과 전투 중간 스냅샷은 이번 범위에서 제외한다.

## 4. 화면 구조

### 타이틀 화면

기존 숲 배경 위에 게임의 결과 패널과 같은 재질의 작은 중앙 패널을 배치한다.

- 게임 제목
- 저장 요약: 캐릭터, 막/노드, 현재 HP
- `이어하기`
- `새 런 시작`

저장이 없으면 저장 요약 대신 `저장된 런이 없습니다`를 표시한다.

### 런 메뉴

상단 진행 정보 영역에 작은 `메뉴` 버튼을 추가한다. 메뉴는 중앙을 계속 가리는 영구 HUD가 아니라 필요할 때만 열리는 모달이다. 모달 뒤에는 어두운 반투명 배경을 적용하고 플레이 입력을 차단한다.

파괴적 행동인 `새 런 시작`은 경고색 보조 버튼으로 구분하며, 확인 단계에서만 최종 실행한다. `Escape`는 메뉴와 확인창을 닫고 현재 런으로 복귀한다.

### 캐릭터 선택

기존 캐릭터 카드의 이름과 HP 옆에 해당 캐릭터의 기존 전투 스프라이트를 `idle` 상태로 표시한다.

- 신규 이미지 자산을 만들지 않는다.
- 화염 격투가, 수호자, 술사, 냉기 기사, 마도기사의 기존 아틀라스와 manifest를 재사용한다.
- 이미지 로드 실패 시 이름과 텍스트 정보만으로도 선택할 수 있어야 한다.
- 데스크톱은 2열, 작은 화면은 1열로 전환한다.
- 카드 전체가 버튼이라는 현재 조작 규칙을 유지한다.
- 키보드 포커스와 접근 가능한 이름에 캐릭터 이름·HP·시작 스킬을 유지한다.

## 5. 컴포넌트 및 상태 경계

### `App`

부팅 모드에 `title`을 추가하고 다음 전환을 소유한다.

- 타이틀 → 저장 이어하기
- 타이틀/런 메뉴 → 캐릭터 선택
- 런 → 타이틀
- 저장 손상 → 복구 화면

저장 파싱과 전투 재개 로직은 하나의 공용 함수로 모아 타이틀과 런 메뉴가 같은 계약을 사용하게 한다.

### `RunGame`

현재 런 세션과 메뉴 열림 상태를 소유한다. 전투와 비전투 화면 모두에 공통 런 메뉴를 렌더링한다. 타이틀 복귀와 새 런 시작은 `App` 콜백으로 위임한다.

### `TitleScreen`

저장 요약과 `이어하기`, `새 런 시작`만 담당한다. 저장 파싱이나 런 생성 로직을 직접 소유하지 않는다.

### `RunMenu`

메뉴 표시, 확인 단계, 키보드 포커스만 담당한다. 실제 저장 삭제·불러오기·화면 전환은 콜백으로 위임한다.

### `CharacterSelect`

캐릭터별 스프라이트 표현 정보를 입력받아 기존 카드에 시각 요소를 추가한다. 런 생성은 계속 상위 컴포넌트가 담당한다.

## 6. 오류 및 예외 처리

- 불러오기 직전에 저장이 사라지거나 손상되면 복구 화면으로 전환한다.
- 저장소 접근 실패는 진행을 막지 않고 경고를 표시한다.
- 새 런 확인 후 저장 삭제가 실패해도 메모리 세션과 캐릭터 선택 진입은 허용한다.
- 스프라이트 로드 실패는 카드 전체 실패로 전파하지 않는다.
- 모달이 열린 동안 전투 카드·동전·턴 종료 입력을 차단한다.

## 7. 접근성 및 반응형 기준

- 타이틀, 런 메뉴, 확인창, 캐릭터 선택은 적절한 dialog/label 구조를 사용한다.
- 모달을 열면 첫 행동으로 포커스를 이동하고 닫으면 메뉴 버튼으로 복귀한다.
- 모든 버튼은 키보드로 실행할 수 있어야 한다.
- 390×844, 900×600, 1280×720에서 가로 스크롤이 없어야 한다.
- `prefers-reduced-motion`에서는 캐릭터 스프라이트를 첫 idle 프레임으로 고정한다.

## 8. 번들 예산 결정

현재 총 번들은 2,726,295바이트로 기존 2.6MiB 한도보다 2바이트만 작다. 기능 추가를 기존 한도 안에 억지로 밀어 넣지 않는다.

- 총량 한도만 최대 16KiB 확장한다.
- JS 400KiB, CSS 70KiB, 단일 파일 700KiB 한도는 유지한다.
- 새 이미지 자산은 추가하지 않는다.
- 실제 구현 후 증가분을 측정하고, 총량 한도는 필요한 값까지만 올린다.

## 9. 테스트 및 완료 조건

- 저장 유무에 따라 타이틀의 `이어하기` 활성 상태가 정확하다.
- 일반 접속은 타이틀을 표시하고 테스트 URL은 기존 빠른 부팅을 유지한다.
- `이어하기`가 저장된 캐릭터·HP·노드 상태를 복구한다.
- 런 메뉴가 전투와 모든 비전투 단계에서 열린다.
- `타이틀로 나가기`는 저장을 보존한다.
- `새 런 시작`은 확인 전 저장을 유지하고 확인 후 저장을 지운 뒤 캐릭터 선택으로 이동한다.
- `마지막 저장 불러오기`는 확인 후 저장 상태를 다시 적용한다.
- 캐릭터 카드마다 올바른 기존 스프라이트가 표시된다.
- 모바일 캐릭터 선택 화면에 가로 스크롤이나 카드 잘림이 없다.
- 저장 손상 복구, 전체 플레이테스트, 피드백, 접근성, 성능 검사를 모두 통과한다.
- 최종 번들이 조정된 총량과 기존 JS·CSS·단일 파일 예산을 통과한다.

## 10. 비목표

- 다중 저장 슬롯
- 전투 턴 중간 저장
- 브라우저 탭 강제 종료
- 신규 캐릭터 일러스트 또는 스프라이트 제작
- 설정·도감·크레딧 등 추가 타이틀 메뉴
