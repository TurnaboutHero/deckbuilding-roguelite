## 0. 요약

PRD v0.3의 MVP(§14)를 **약 9~12주(1인 기준)** 에 걸쳐 7개 마일스톤(M0~M6)으로 구현한다.

핵심 전략 3가지:

1. **헤드리스 코어 우선** — 게임 규칙 전체를 UI 의존성 없는 순수 TypeScript 패키지로 먼저 구현한다. 규칙 조합(면 판정 × 취급 태그 × 소모 영역 × 임시 코인)은 버그의 온상이므로 결정론 테스트와 시뮬레이터로 굳힌 뒤 UI를 얹는다.
2. **손맛 게이트를 4~5주차에 강제** — MVP의 목표는 "배치 후 플립" 문법의 재미 검증(PRD §14.1)이다. 코어 우선이되, 플립 연출을 포함한 최소 UI 플레이테스트를 M2 종료 시점에 반드시 수행하고, 신호가 나쁘면 콘텐츠 확장 전에 루프를 수정한다.
3. **측정 2단 구조** — 봇 시뮬레이션은 밸런스 사전 검증(회귀 게이트)으로만 쓰고, PRD §16.2 성공 지표의 최종 판정은 사람 플레이테스트 로그로 한다.

PRD가 미정의한 규칙은 §4에서 전부 결정했고, v1.0의 미확정 항목은 PRD v0.3 반영으로 **모두 확정**됐다(§11 확정 내역). 구현은 이 문서를 명세로 삼는다.

## 1. 목표와 범위

### 1.1 목표 (PRD §14.1)

전체 콘텐츠가 아니라 **"코인을 스킬에 배치하고 플립하는 전투 문법이 재미있는지"** 를 검증한다. 이 문서의 모든 우선순위 판단은 이 한 문장에 종속된다.

### 1.2 범위 (PRD §14.2~14.5)

- **포함**: 전사 1종(기본 8 + 화염 2), MVP 스킬 6종(베기/방어/불타는 일격/점화/점화 검술/화염 폭주), 적 3종(공격형/방어형/디버프형), 기본·화염 코인, 임시 코인, 5영역 코인 순환, 소비형 스킬, 취급 태그, 최소 보상, 선형 런.
- **선택 포함**: 마나 코인 (M5 보상 풀에서 여유 시).
- **제외**: 상점, 이벤트, 보스, 맵, 다른 캐릭터, 냉기/전기/혈액 속성. 단 데이터 모델은 확장을 막지 않게 설계한다(§6).

## 2. 기술 스택

| 영역 | 선택 | 근거 |
|---|---|---|
| 언어 | TypeScript (strict) | 코어 로직·콘텐츠·UI·시뮬레이터를 단일 언어로. 효과 데이터의 타입 검증(`satisfies`)이 공짜 |
| 코어 | 순수 TS 패키지 (프레임워크 의존 0) | 결정론 테스트, 헤드리스 시뮬레이션, 이후 엔진 이식(Godot/Unity) 시에도 규칙 명세로 재사용 가능 |
| UI | Vite + React | 이 게임은 보드/카드형 UI 중심(StS류) — DOM이 캔버스 엔진보다 프로토타이핑 속도가 빠름. 플립 연출은 CSS 3D로 충분 |
| 테스트 | Vitest (+ fast-check property 테스트) | 모노레포 통합, 워치 모드 |
| 시뮬레이터 | 자체 CLI (tools/sim) | PRD §16.2 정량 지표 측정, 밸런스 A/B |
| 패키지 관리 | pnpm workspace | 코어/콘텐츠/UI/심 분리 |
| CI | GitHub Actions | typecheck + lint + test + 스모크 시뮬 |

> 웹 선택의 트레이드오프: 상용 배포 시 엔진 전환 가능성이 있으나, MVP 목표가 "문법 재미 검증"이므로 반복 속도가 최우선이다. 코어를 순수 데이터+리듀서로 유지하면 규칙은 이식 자산으로 남는다.

## 3. 아키텍처

### 3.1 모노레포 구조

의존 방향은 단방향 고정: `content → core`, `ui → core + content`, `sim → core + content`. core는 DOM/React/Node API를 import하지 않는다(린트로 강제).

```
deckbuilding-roguelite/
├─ pnpm-workspace.yaml
├─ docs/                       # PRD.md, implementation-plan.md
├─ packages/
│  ├─ core/                    # @game/core — 헤드리스 게임 로직
│  │  └─ src/
│  │     ├─ ids.ts             # branded id 타입
│  │     ├─ rng.ts             # 결정론 RNG + 스트림 파생
│  │     ├─ content-types.ts   # CoinDef/SkillDef/EnemyDef/CharacterDef 스키마
│  │     ├─ combat/
│  │     │  ├─ state.ts        # CombatState, 영역(zone), 유닛
│  │     │  ├─ commands.ts     # Command union + 검증
│  │     │  ├─ reducer.ts      # step(state, cmd) → {state, events}
│  │     │  ├─ resolve/        # 효과 해석기 (flip / consume / status)
│  │     │  ├─ enemy.ts        # 의도 결정 + 적 페이즈
│  │     │  ├─ preview.ts      # 면 조합 전수 열거 프리뷰 (§3.5)
│  │     │  └─ events.ts       # CombatEvent union
│  │     ├─ run/               # RunState, 전투 진행, 보상
│  │     └─ serialize.ts       # 세이브/리플레이 스키마
│  └─ content/                 # @game/content — TS 리터럴 데이터
│     └─ src/                  # coins / skills / enemies / characters
├─ apps/
│  └─ ui/                      # @game/ui — Vite + React
└─ tools/
   └─ sim/                     # @game/sim — 시뮬레이터 CLI (봇, 지표 집계)
```

- content 분리 이유: 밸런스 수정이 로직 diff와 섞이지 않게 하고, sim이 "동일 로직 + 다른 수치"를 주입해 A/B 실험을 하기 위함. core는 콘텐츠를 import하지 않고 `ContentDb`를 인자로 받는다.

### 3.2 헤드리스 코어 원칙 — 순수 리듀서 + 이벤트 로그

```ts
step(state: CombatState, cmd: Command, db: ContentDb): StepResult
// StepResult = { ok: true, state, events: CombatEvent[] } | { ok: false, error }
```

1. **결정론·리플레이가 공짜**: `초기 시드 + Command[]`가 전투의 완전한 기록. 버그 재현, 골든 테스트, 시뮬레이터가 전부 이 위에 선다.
2. **이벤트가 단일 진실 소스**: UI 애니메이션 큐와 sim 지표 집계가 **같은** `CombatEvent[]`를 소비한다. 이벤트는 과거형 사실이며 id·숫자만 담아 100% 직렬화 가능.
3. **성능 비문제**: 전투 4~7턴, 코인 ~15개 규모. 구조 공유 복사로 sim에서 초당 수천 판이 나온다.
4. UI는 규칙을 전혀 모른다 — "배치 가능한가?"조차 `legalCommands()`/dispatch 실패로 판정한다.

### 3.3 콘텐츠 표현 — TS 리터럴 + 효과 AST

스킬 효과는 코드 함수가 아닌 **데이터(효과 원자 EffectAtom 배열)** 로 표현하고 코어가 해석한다.

- 코드 함수 방식을 배제하는 이유: ① UX 필수 요구(§15 "배치 후 예상 효과")와 sim의 EV 계산이 효과를 *실행하지 않고 해석*해야 하는데 불투명 함수는 이를 막는다. ② 직렬화/리플레이와 궁합이 나쁘다. ③ 밸런스 수치 수정이 코드 수정이 된다.
- JSON이 아니라 TS 리터럴 + `satisfies`: 타입 체크·자동완성을 공짜로 얻고 로더/스키마 검증 코드가 불필요하다.
- 탈출구: 표현 불가능한 스킬용 `{ kind: 'custom', key }` 원자를 예약하되, **MVP 6종은 전부 기본 원자로 표현 가능하므로 MVP에서 custom 사용 0개를 규율로 삼는다.**
- 효과 설명 문구는 `EffectAtom → 한국어 문자열` 렌더러 하나로 자동 생성한다(설명 텍스트와 실제 효과의 불일치 방지).

### 3.4 RNG 설계 — 시드 계층 + 스트림 분리

작고 빠른 PRNG(mulberry32 또는 xoshiro128\*\*) + 라벨 해시 파생.

```
runSeed ("BRAVE-EMBER-42")
 ├─ derive(run, 'reward', combatIndex)      # 보상 롤
 └─ derive(run, 'combat', combatIndex, attempt)   # 전투 마스터 시드
     ├─ derive(combat, 'flip')              # 코인 플립 전용
     ├─ derive(combat, 'shuffle')           # 셔플 전용
     └─ derive(combat, 'ai')                # 적 의도 전용
```

스트림 분리 이유:

1. **교차 오염 차단** — 플레이어가 스킬을 하나 더 써서 플립을 더 소비해도 다음 셔플·적 의도가 변하지 않는다. 단일 스트림이면 "행동 순서로 적 패턴 낚시"가 성립하고, sim의 "같은 시드, 다른 전략" 비교가 무의미해진다.
2. **테스트 격리** — flip 스트림에만 스크립트 더블(`scriptedFlips(['H','T'])`)을 주입해 특정 면 조합을 강제하는 테스트가 가능하다.
3. **전투 재입장 세이브스컴 차단** — 전투 시드에 시도 횟수(`attempt`)를 소금으로 넣어, 중도 이탈 후 재시작 시 같은 전투라도 드로우/의도가 달라지게 한다. (스트림이 행동과 독립이라는 설계 자체가 "리허설 후 재도전" 악용을 열 수 있으므로 이 소금이 필수다.)

규율: core 내 `Math.random` 사용 금지(린트 강제). RNG 상태는 CombatState에 내장되어 상태와 함께 직렬화된다.

### 3.5 코어-UI 경계와 프리뷰

```ts
interface CombatSession {
  state(): CombatState;
  legal(): Command[];
  preview(cmd: Command): Preview | null;   // §15 "배치 후 예상 효과"
  dispatch(cmd: Command): StepResult;
  commandLog(): readonly Command[];
}

interface Preview {
  // flip 스트림을 강제 면으로 스텁하고 2^n 분기(n = 배치 코인 수 ≤ 5, A18 상한) 전수 실행
  branches: { faces: Face[]; probability: number; events: CombatEvent[] }[];
  // 효과 축별 집계 — "앞면=고점" 같은 고정 라벨 금지 (방어는 뒷면이 고점, PRD §8.2/§13.1)
  byAxis: { damage: MinMax; block: MinMax; heal: MinMax;
            statuses: Partial<Record<StatusId, MinMax>> };
  expected: { damage: number; block: number /* … 축별 기대값 */ };
}
```

- **프리뷰는 면 조합 전수 열거로 계산한다.** "뒷면 전부=저점 / 앞면 전부=고점" 같은 스칼라 접기는 금지 — 방어(뒷면 +3)와 점화(앞면 화상 vs 뒷면 피해)처럼 축이 다르거나 뒷면이 고점인 스킬에서 틀린다.
- 봇의 기대값 평가도 같은 분기 열거를 재사용한다(스킬당 최대 2⁵=32분기 — A18 코스트 상한 5 기준, 여전히 비용 무시 가능).
- 적 의도 대비 표시 시 방어 초기화 시점(D3)을 반영해 "이번 적 행동에 실제로 남을 방어"를 보여준다.

### 3.6 직렬화 / 세이브

모든 코어 상태가 plain data이므로 직렬화는 구조적으로 공짜. 정책만 결정한다:

- **저장은 전투 경계에서만.** 전투가 4~7턴으로 짧으므로 중도 이탈 시 해당 전투는 재시작(§3.4의 attempt 소금 적용). 전투 중 스냅샷 저장은 MVP 유지비 대비 가치가 없다.
- 커맨드 로그는 **버그 리포트/리플레이 파일**로 항상 기록한다: `{ version, contentVersion, runSeed, combatIndex, commands[] }`.
- 파생 가능한 것(더미 순서, 적 의도)은 저장하지 않는다 — `runSeed + combatIndex + attempt`에서 재파생.

### 3.7 과설계 가드레일 (MVP = 전투 문법 재미 검증)

| 만들지 않는 것 | 대신 |
|---|---|
| 범용 상태이상 훅/키워드 엔진 | 화상·방어를 엔진 고정 로직으로 직접 구현. 동상/전기 도입 때 일반화 |
| 턴 버프(트리거) 서브시스템 | MVP 스킬 6종에 소비자 없음(화염검·불의 심장은 비MVP). 훅 자리만 예약 (§12) |
| ECS, 이벤트 버스, DI 컨테이너 | 리듀서 + 이벤트 배열 return |
| JSON 콘텐츠 파이프라인/에디터 | TS 리터럴 + `satisfies` + 콘텐츠 검증 테스트 |
| 맵/상점/이벤트 시스템 | 선형 `combatIndex` 진행 + 최소 보상 화면 |
| 애니메이션 타임라인 프레임워크 | 이벤트 순차 재생 큐 (플립 연출만 집중 투자) |
| 콤보 스캐너·반사실 리플레이·CI 밸런스 밴드 | 스킬 10종+ 확장 시점의 백로그 (§8.3, §12) |
| `custom` 효과 원자 사용 | MVP 스킬 6종은 기본 원자만으로 구현 |

```
  T0  스킬 사용 상태 리셋 — 전 슬롯 usedThisTurn = false,

    플레이어가 "턴 종료" 선택 or 선택지 없음 → break
    스킬 선택 → 배치/연료·대상 지정(취소 자유) → 사용 선언(취소 불가)
    → RESOLVE_FLIP / RESOLVE_CONSUME (즉시 해결)
    전투 종료 시 return

PLAYER_TURN_END:
  E0  미선언 배치 코인 → 손패 복귀                         (A16)
  E1  플레이어 화상 틱: 스택만큼 방어 무시 피해 → 패배 판정 → 스택 −1  (D2)
  E2  (예약) 턴 버프 일괄 소멸 — MVP 소비자 없음
  E3  전 영역 코인의 취급 태그 만료                        (D6)
  E4  손패 전체 → 버림 더미                                (D7)

ENEMY_PHASE:
  X1  각 생존 적: 방어 = 0                                 (D3: 적은 자기 행동 직전 초기화)
  X2  인덱스 순으로 공개된 의도 그대로 실행 (재추첨 없음) → 패배 판정
  X3  각 생존 적 화상 틱 → 승리 판정                       (화상만으로 전멸 가능)
  X4  생존 적 다음 의도 결정(ai 스트림) 후 공개             (D8)
  → PLAYER_TURN_START
```

순서 근거: 화상 틱(E1)이 태그 만료(E3)보다 앞이어야 "이번 턴" 효과의 적용 범위에 모호성이 없고, 손패 정리(E4)가 마지막이어야 태그 만료가 어느 영역이든 일괄 처리된다.

### 5.2 플립형 스킬 해결 (RESOLVE_FLIP)

```
P0 검증      이번 턴 스킬 사용 수 < 3(D0), 스킬 이번 턴 미사용(D0),
             일회성이면 이번 전투 미사용(A17),
             배치 수 == cost, 코인들이 손패 소속,
             대상 필요 시 primaryTarget 생존. 대상 없는 스킬의 공격성 속성 효과용
             보조 타겟 = 첫 번째 생존 적 (A8)
P1 커밋      코인 손패 → 스킬 슬롯, usedThisTurn 마킹,
             skillUsesThisTurn +1, 일회성이면 usedThisCombat 마킹
             (해결 중 재진입 차단 — 효과 적용 전에 전부 기록)
P2 플립      슬롯 순서대로 각 코인 flip 스트림에서 1회 → 면 기록
             (반드시 전부 플립 후 효과 — perHead가 전체 개수를 읽어야 함)
P3 스냅샷    각 코인의 (면, 유효속성) 고정. headCount/tailCount 계산.
             생존 적 목록 스냅샷. 이후 손패가 변해도 이번 해결의 판정 불변
P4 효과 수집  스킬 lines 순회 (순수 계산):
               always → 발동 | face any → 해당 면 ≥1이면 1회
               face per → 수치 효과는 amount × 개수로 합산, 생성 효과는 개수만큼 반복
             동일 대상 피해는 하나의 "피해 패킷"으로 합산
             (스킬 1회 사용 = 1히트. 베기 앞면은 10짜리 한 방이지 6+4 두 방이 아님)
P5 스킬 효과  패킷/원자를 텍스트 순서로 적용. 각 원자 직후 사망·승패 마이크로 판정 (A6)
             코인 생성·취급 태그 부여도 여기서 실행
P6 속성 효과  슬롯 순서대로 코인 개별 판정 (D1 후단):
             유효속성의 정의된 면 == 그 코인의 면이면 발동 (화염 앞면 → 화상 +1, 코인마다)
             대상이 이미 사망했으면 불발
P7 (예약)    턴 버프 발동 지점 — attack 태그 스킬 해결 시. 피해 발생 여부와 무관하게
             태그만으로 판정 (화염검 "사용할 때마다"). MVP에서는 no-op
P8 최종 판정  승패 확정 시 잔여 중단
P9 정리      슬롯 코인 → 버림 더미. 면 기록 소멸. 취급 태그는 유지(E3에서만 만료)
```

### 5.3 소비형 스킬 해결 (RESOLVE_CONSUME)

```
C0 검증      이번 턴 스킬 사용 수 < 3(D0), 스킬 이번 턴 미사용(D0),
             일회성이면 이번 전투 미사용(A17),
             지정 코인 수 == 조건 수 (1~3, A18),
             각 코인이 손패 소속 && 유효속성 == 요구 속성 (취급 태그 인정, D6)
C1 소비      코인 손패 → 소모 영역 (D5). 플립 없음 → 면·속성 효과 완전 미발동 (§8.3)
             usedThisTurn 마킹, skillUsesThisTurn +1, 일회성이면 usedThisCombat 마킹
             생존 적 스냅샷은 이 시점에 고정 (AoE 대비)
C2 확정 효과  텍스트 순서로 적용 + 마이크로 판정
C3 (예약)    턴 버프 발동 지점 — P7과 동일 규칙. 자기 자신이 등록한 버프는
             이후 해결되는 스킬부터 유효 (소급·자가 발동 없음)
C4 최종 판정
```

### 5.4 취급 태그 (용광로 / 화염 폭주)

- 모델: 코인 인스턴스의 속성 부여 태그. 모든 속성 판정은 `유효속성 = 기본속성 ∪ 부여태그` 단일 함수를 경유한다.
- 화염 폭주(`allBasicInHand`): **해결 시점 손패 스냅샷**에 스탬프. 오라가 아님 — 이후 유입 코인 미적용, 배치되어 슬롯에 간 코인(폭주 코스트 포함) 미적용 (A14).
- 손패에 기본 코인이 0개면 해당 라인만 불발하고 스킬은 사용 가능("스킬은 실패하지 않는다").
- 태그 코인이 턴 내 리셔플로 뽑을 더미에 돌아가도 유지 — 어차피 E3에서 만료되므로 악용 불가.
- 중복 부여: 집합에 추가 (C3 다속성 대비).

### 5.5 임시 코인 수명주기

- 생성: 효과 원자(P5/C2). `isTemporary: true`, 새 인스턴스 id. 목적지는 스킬 데이터가 지정 — §10 강도 사다리(버림 더미 < 뽑을 더미 < 손패)를 데이터로 표현.
- **뽑을 더미 추가는 무작위 위치 삽입**(shuffle 스트림) — 맨 위 삽입은 확정 드로우가 되어 §10 등급을 깨뜨린다.
- 순환: 전투 중에는 일반 코인과 완전히 동일(파괴가 아니라 버림 더미행 — 이번 전투 내내 자산). 제거는 전투 종료 정산 단 한 번, 전 영역 대상.

### 5.6 사망·승패

- 모든 원자 효과 직후 마이크로 판정. 패배 우선(순서 고정 = 결정론).
- 사망한 대상을 향한 같은 해결 내 잔여 효과는 조용히 불발 (예: 피해로 사망 후 P6 화상).
- 확정 시 잔여 효과·페이즈 중단 → 종료 루틴(임시 코인 제거, 태그/상태 소멸, 승리 시 보상).
- 엣지: 폭주 뒷면 자해로 플레이어 사망 → P5 도중 즉시 패배(앞면 라인이 남아 있어도 중단).

### 5.7 엣지 케이스 체크리스트 (전부 테스트 대상)

| 케이스 | 규칙 |
|---|---|
| 드로우 시 양 더미 합계 < 5 | 부분 드로우, 에러 아님 |

### 5.8 MVP 6종 골든 트레이스 (테스트 정답지)

셋업: 신규 대상 적 (방어 0), 플립은 스크립트 강제.

| 스킬 | 배치 | 플립 | 기대 결과 |
|---|---|---|---|
| 베기 | 기본 1 | H / T | 피해 10 / 피해 6 |
| 베기 | 화염 1 | H / T | 피해 10 + 화상 1 / 피해 6 |
| 방어 | 기본 1 | H / T | 방어 5 / 방어 8 |
| 불타는 일격 | 기본 2 | HH / HT / TT | 피해 14 / 11 / 8 — 공통: 임시 화염 1개 버림 더미 |
| 불타는 일격 | 화염 2 | HH | 피해 14 + 화상 2 (속성은 코인별, D1) |
| 점화 | 화염 1 | H / T | 화상 3 (기본1+앞면1+코인proc1) / 화상 1 + 피해 3 |
| 점화 검술 | 화염 1 소비 | 플립 없음 | 피해 10 + 화상 2, 코인 → 소모 영역 |
| 화염 폭주 | 기본 1 | H / T | 손패 기본 전부 화염 태그 + 임시 화염 1 손패 / 태그 + 자신 피해 2 |

D1 경계 골든: 불타는 일격 HH = +3×2 (`per`) vs 가상의 "앞면: 피해 +4" 2코인 스킬 HH = +4 1회 (`any`) — 두 케이스를 나란히 고정. 판정 단위는 PRD v0.3 §8.1로 확정됨. 강타 HH = +5×2 (`per`)도 보상 풀 도입 시(T5.4) 이 표에 추가.

## 6. 데이터 모델 (핵심 타입)

```ts
// ---- 코인 ----
type Face = 'heads' | 'tails';
type Element = 'fire' | 'mana' | 'frost' | 'lightning' | 'blood';

interface CoinDef {
  id: CoinDefId;                    // 'basic' | 'fire' | 'mana'
  element: Element | null;          // null = 기본 코인
  proc?: { face: Face; effects: EffectAtom[] };  // 화염: heads → burn 1
}

interface CoinInstance {
  uid: CoinUid;                     // 전투 내 고유 (총량 원장 추적용)
  defId: CoinDefId;
  permanent: boolean;               // 6.6장 — 전투 종료 정산과 UI 표기
  grants: Element[];                // D6 취급 태그 (턴 종료 만료, 집합형 — C3 대비)
}
// 유효속성(coin) = def.element ∪ grants — 모든 속성 판정의 단일 진입점

// ---- 스킬 ----
type SkillDef = FlipSkillDef | ConsumeSkillDef;

interface SkillDefBase {
  id: SkillId; name: string;
  rarity: 'common' | 'advanced' | 'rare';
  tags: readonly ('attack' | 'defense' | 'utility'
                 | 'ultimate')[];   // A9. 'ultimate'는 A18 lint용 예약 (비MVP)
  targetType: 'single-enemy' | 'all-enemies' | 'self' | 'none';  // A7
  oncePerCombat?: boolean;          // A17 일회성 — 전투당 1회 (기본 false)
}

// ---- 캐릭터 (B18) ----
interface CharacterDef {
  id: CharacterId; name: string;
  maxHp: number;                            // 전사 70
  startingBag: CoinDefId[];                 // 기본×8 + 화염×2
  startingSkills: SkillId[];                // 슬롯 6 중 4 장착
  trait: {                                  // 고유 특성 — 패시브 1개 (PRD v0.4 §7)
    id: string; name: string;               // 전사: 'ember-pouch' 불씨 주머니
    hook: 'combatStart' | 'turnStart';      // 자동 발동 조건 (확장: onConsume 등)
    effects: EffectAtom[];                  // 불씨: addCoin fire → draw (임시)
  };
}
interface FlipSkillDef extends SkillDefBase {
  type: 'flip'; cost: number;
  base: EffectAtom[];
  heads?: { mode: 'any' | 'per'; effects: EffectAtom[] };   // D1/B12
  tails?: { mode: 'any' | 'per'; effects: EffectAtom[] };
}
interface ConsumeSkillDef extends SkillDefBase {
  type: 'consume';
  consume: { element: Element; count: number };
  effects: EffectAtom[];
}

// ---- 효과 원자 (MVP 세트) ----
type EffectAtom =
  | { kind: 'damage'; amount: number }
  | { kind: 'block'; amount: number }
  | { kind: 'selfDamage'; amount: number }
  | { kind: 'applyStatus'; status: StatusId; stacks: number; to: 'target' | 'self' }
  | { kind: 'addCoin'; coin: CoinDefId; zone: 'draw' | 'discard' | 'hand'; count: number }
      // 생성 코인은 항상 임시 — §10 원칙을 타입으로 강제 (permanent 옵션 자체가 없음)
  | { kind: 'grantElement'; element: Element; scope: 'allBasicInHand' };
// 예약(비MVP): heal, damageAll, grantElement chooseBasicInHand(용광로),
//              addTurnTrigger(화염검류), 면 조건 allHeads/allTails,
//              스킬 자체의 속성 조건 라인(§8.1 "속성:"), custom

// ---- 전투 상태 ----
interface CombatState {
  turn: number;
  phase: 'player' | 'enemy' | 'victory' | 'defeat';
  player: { hp: number; maxHp: number; block: number;
            statuses: Partial<Record<StatusId, number>>;
            nextDrawPenalty: number };                       // B5 위축
  enemies: EnemyState[];            // { defId, hp, maxHp, block, statuses, intent }
  coins: Record<CoinUid, CoinInstance>;
  zones: { draw: CoinUid[]; hand: CoinUid[];
           placed: Record<SlotId, CoinUid[]>;
           discard: CoinUid[]; exhausted: CoinUid[] };       // D5 소모 영역
  slots: { skillId: SkillId; usedThisTurn: boolean;
           usedThisCombat: boolean }[];                      // D0 + A17 일회성
  skillUsesThisTurn: number;                                 // D0 턴당 총 3회 카운터
  rng: { flip: RngSnapshot; shuffle: RngSnapshot; ai: RngSnapshot };
  nextUid: number;
}

// ---- 커맨드 / 이벤트 ----
type Command =
  | { type: 'placeCoin'; coin: CoinUid; slot: SlotId }
  | { type: 'unplaceCoin'; coin: CoinUid }
  | { type: 'useFlipSkill'; slot: SlotId; target?: number }
  | { type: 'useConsumeSkill'; slot: SlotId; coins: CoinUid[]; target?: number }
  | { type: 'endTurn' };

type CombatEvent =   // 전체 목록은 events.ts — 대표만 발췌
  | { type: 'coinsDrawn'; coins: CoinUid[] }
  | { type: 'coinFlipped'; coin: CoinUid; face: Face }
  | { type: 'skillUsed'; slot: SlotId; skill: SkillId; kind: 'flip' | 'consume' }
  | { type: 'damageDealt'; target: TargetRef; amount: number; blocked: number;
      source: 'skill' | 'burn' | 'enemy' | 'self' }
  | { type: 'blockGained'; target: TargetRef; amount: number }
  | { type: 'blockCleared'; target: TargetRef; amount: number }   // D3 — 양측 구분
  | { type: 'statusApplied' | 'statusTicked'; target: TargetRef; /* … */ }
  | { type: 'coinCreated' | 'coinsConsumed' | 'elementGranted'; /* … */ }
  | { type: 'intentRevealed'; enemy: number; intent: EnemyIntent }
  | { type: 'combatEnded'; result: 'victory' | 'defeat'; turns: number };
```

`endTurn` 하나가 E0~E4 + 적 페이즈 전체 + 다음 턴 시작까지 진행한다(MVP에서 적 턴에 플레이어 입력이 없으므로 phase를 쪼개 커맨드를 늘릴 이유가 없음). 애니메이션 분절은 이벤트 스트림이 담당한다.

## 8. 테스트 전략

### 8.1 단위·골든·불변식

- **골든 테스트**: §5.8 표를 `{셋업, 강제 플립, 기대 이벤트}` 픽스처로. 이벤트 스트림 단언(상태 diff보다 회귀 원인 추적이 쉬움).
- **결정론 테스트**: 같은 시드+커맨드 → 상태·이벤트 완전 일치(직렬화 라운드트립). 골든 리플레이 파일을 CI에서 재실행.
- **영역 불변식** (property-based, 퍼즈·시뮬레이터에 상시 삽입):
  1. **총량 원장**: Σ(전 영역) = 초기 10 + `coinCreated` 이벤트 수. 코인 uid 중복/증발 0.
  2. **소모 영역 격리**: 리셔플 시 소모 영역 코인 복귀 0 (D5).
  3. **임시 코인 수명**: 전투 종료 후 전 영역에서 임시 코인 0, 영구 구성 전투 전후 동일.
  4. **턴 종료 정리**: 손패·슬롯 공집합 (D7).
  5. **유계**: 스택 ≥ 0, HP ≤ maxHP, 방어 ≥ 0.

### 8.2 시뮬레이터

```
정책 인터페이스: plan(state) → Command   (한 커맨드씩 — legalCommands 기반)
에피소드 시드 = derive(runSeed, episode), NDJSON 이벤트 로그 → 집계기
규모: 셀당(정책×적×수치 세트) 2,000판. 수치 A/B는 공통 난수(CRN)로 분산 축소
```

| 정책 | 규칙 |
|---|---|
| RandomBot | 무효 행동 상한 캘리브레이션 + 지표 파이프라인 검증 |
| AggroBot | 피해 EV 내림차순 배치, 방어는 의도 피해 ≥ HP 25%일 때만 |
| TurtleBot | 적 공격 의도 시 방어 우선(D0 제약상 턴당 최대 1회), 잔여는 저위험 공격 |
| GreedyEVBot | 1턴 전방 탐색 — 배치 조합 열거 + 프리뷰 분기 기대값, 화상 한계가치 × 잔여 턴 추정 포함 |

관측: 정책 간 승률 격차가 전략 깊이의 대리 지표(GreedyEV ≫ Random이어야 하고, GreedyEV ≈ Aggro면 의사결정이 얕다는 신호). D0 때문에 MVP 방어 수단이 방어 스킬 1종(턴당 상한 8)뿐이라 Turtle/Aggro 격차가 작을 것 — 마나 코인·방어 스킬 추가 후에 재검.

### 8.3 지표 측정 — 2단 구조

**1단(봇 시뮬) = 밸런스 사전 검증. 2단(사람 플레이테스트) = PRD §16 최종 판정.** 봇 지표로 사람의 재미·지각 지표를 판정하지 않는다(특히 무효 행동률은 봇 설계상 거의 자동 충족이라 검증력이 없고, "화염 코인을 더 얻고 싶다"는 욕구 지표는 기계적 우위 A/B와 측정 대상이 다르다).

| 지표 (PRD §16.2) | 조작적 정의 | 판정 주체 |
|---|---|---|
| 평균 턴 수 4~7 | 개시된 플레이어 턴 수. 전체 평균 ∈ [4,7], 적별 ∈ [3,7.5] (주술사는 "빠른 처치" 콘셉트상 하한 3 허용) | 1단 예비 → 2단 확정 |

## 9. 초기 밸런스 수치

모든 수치는 시뮬레이터 튜닝(T6.4)의 **출발점**이다. 산출 근거를 남겨 튜닝 시 노브를 명확히 한다.

### 9.1 스킬 EV (공정 코인, 신규 대상, 화상 풀 틱 가정)

| 스킬 | 코인 | 직접 피해 EV | 화상 가치 | 총 EV | 비고 |
|---|---|---|---|---|---|
| 베기 | 기본 1 | 6+0.5×4 = **8** | — | 8 | 피해 효율 기준선 |
| 베기 | 화염 1 | 8 | +0.5 | **8.5** | 코인 proc 앞면 50% |
| 방어 | 1 | 방어 5+0.5×3 = **6.5** | — | 6.5 | D0 → 턴당 방어 상한 8 |
| 불타는 일격 | 2 | 8+1×3 = **11** | — | 11+α | α = 임시 화염 코인 가치 ≈ 2.5 |
| 강타 (보상 풀) | 2 | 8+1×5 = **13** | — | 13 | 코인 생성 없는 순수 딜러 — 일격(11+α)과 등가 근처, 분산은 더 큼(범위 8~18) |
| 점화 | 기본 1 | 1.5 | 2 | **3.5** (스택 s당 +1.5s) | s=3에서 베기와 동률, **s≥4부터 추월** |
| 점화 | 화염 1 | — | — | 2s+5 | |
| 점화 검술 | 화염 1 소비 | **10 확정** | 2s+3 | **2s+13** | 무플립 확정 |
| 화염 폭주 | 1 | −1(자해 EV) | 전개 의존 | 콤보 턴 +10~15 | §9.4 |

- **화염 코인의 한계 가치 α**: 점화 검술로 소비하면 13, 베기 코스트로 쓰면 8.5 → 순이익 +4.5. 순환 지연(10코인 주머니 = 2턴 주기)·뒷면 확률 할인 → **α ≈ 2.5~3**. 이 값이 "화염 코인을 더 얻고 싶다"(§16.1)의 수학적 토대다.
- 점화는 신규 대상에겐 약하고(3.5 < 8) 스택이 쌓이면 강해진다 — "화상 누적 빌드"가 수치로 성립. 단 s≥4 도달에는 화염 공급(불타는 일격·폭주) 시너지가 필요.
- 표준 턴 (v1.2 — 턴당 3스킬 제한 반영, 시작 키트 기준):
  - 공격 지향: 베기(1)+불타는 일격(2)+점화(1) = 4코인 3스킬 → 직접 피해 EV **20.5/턴** + 화상 ~2스택 ≈ 실효 ~22/턴, 방어 0
  - 혼합: 방어(1)+베기(1)+불타는 일격(2) = 4코인 3스킬 → 피해 19 + 방어 6.5
  - 시작 키트(1/1/2/1코스트)로는 **매 턴 코인 1개가 자연 낭비**됨 — 2~3코스트 스킬 채용 유인이자 코인 낭비율 관측 대상. v1.1의 4스킬 표준 턴(방어+베기+일격+점화)은 3회 제한으로 불가능해짐

### 9.2 화상 수학 (D2: 스택만큼 피해 후 1 감소)

n스택 1회 부여, 잔여 틱 k회일 때 총 피해 = k≥n이면 n(n+1)/2, 아니면 kn − k(k−1)/2:

| n \ k | 1 | 2 | 3 | 4 | 풀 틱 |
|---|---|---|---|---|---|
| 1 | 1 | 1 | 1 | 1 | **1** |
| 2 | 2 | 3 | 3 | 3 | **3** |
| 3 | 3 | 5 | 6 | 6 | **6** |
| 4 | 4 | 7 | 9 | 10 | **10** |
| 5 | 5 | 9 | 12 | 14 | **15** |

- **한계 가치의 초선형성**: 스택 s에서 +1의 한계 피해 = min(s+1, k). 화상은 쌓을수록 스택당 가치가 커진다 — 방어형 적 설계(화상 보상)와 폭주 리스크의 수학적 뿌리.
- 4~7턴 전투에서 k=2~4가 일반적 → 표의 절단 열이 실전값. 마지막 턴 부여 화상은 가치 절반 이하 → "틱 0회 화상"을 무효 행동 지표에 포함한 이유.

### 9.3 유닛 수치 초안

**플레이어 시작 HP: 70.** 근거: 턴당 방어 상한 8 < 공격형 내려찍기 11 → 턴당 칩 피해 3~4 확정 설계. 전투당 손실 12~18(HP의 17~26%) → 회복 없이 일반 전투 3~4회 유지, 연속 저점에서만 사망권.

| 적 | HP | 행동 사이클 (의도 공개) | 설계 근거 · 예상 턴 |
|---|---|---|---|
| 공격형 "약탈자" | **75** | 내려찍기 11 → 쌍격 4×2 → 내려찍기 11 (평균 10/턴) | 11 > 방어 상한 8 → 방어해도 관통, "방어 강제 + 시간 압박". ~22/턴(v1.2) → **4~5턴** |
| 방어형 "수문장" | **70** | 방벽 8 + 타격 5, 3턴마다 강화 방벽 12 (방어는 자기 행동 직전 리셋 후 재획득, D3) | 직접 피해만 12.5/턴 → ~6턴 / 기본 화상 포함 ~15/턴 → ~5턴 / **화상 빌드 시 4~5턴** — "화상 누적의 가치"를 턴 수 차이 1~1.5로 정량화 |
| 디버프형 "주술사" | **60** | 위축(다음 턴 드로우 −1) → 타격 9 → 반복 (위축은 격턴) | 방치 시 평균 출력 ≈ (22+17.5)/2 ≈ 19.8(v1.2) → ~3턴. 집중 공격(~25/턴) 시 **2~3턴** — "빠른 처치 or 방어 운영" 양자택일. 하한 3턴 허용(§8.3) |
| 런 구성 | — | 전투 1~5: 약탈자 → 주술사 → 수문장 → 약탈자+ → 수문장+ (+ = HP/피해 10~15% 강화) | 후반 2전투는 보상 성장 검증용 |

예측 전체 평균 ≈ 4.5턴(v1.2 출력 기준, 목표 대역 내). 실측이 4 미만이면 1차 노브 = 전 적 HP +10%, 2차 노브 = 방어 스킬 수치.

> v1.2 주의: 턴당 3회 제한으로 턴 출력이 소폭 하락(~23 → ~20-22/턴, §9.1). 예상 턴 수는 대역 내로 추정되나(약탈자 ~4-5턴), **M6 시뮬 1순위 재검 항목**이다. 불씨 주머니(B18)는 화상 경로를 소폭 강화해 일부 상쇄.

### 9.4 한 턴 폭발 상한 검증 (v1.2 — 턴당 3회 제한 + 폭주 일회성 반영)

턴당 3회 제한(D0)으로 상한 조합은 "스킬 3개"로 고정된다. 폭주는 스킬 슬롯 1개를 변환에 쓰므로 **상한 조합은 오히려 폭주 없는 순수 딜 3연타**다 (폭주는 상한이 아니라 화염 proc/연료 공급으로 평균을 올리는 스킬).

| 조합 (스킬 3개) | 최대 즉발(첫 화상 틱 포함) | 성립 조건 · 확률 |
|---|---|---|
| MVP 6종: 점화 검술 + 일격 HH + 베기 H | 10+14+10 = 34 + 화상 4 = **38** (검술 소비 후 장전 가능 화염은 2개 → 화상 = 확정 2 + proc 2) | 손패 화염 3개(불씨 포함 6.1%) × 1/8 — 극히 드묾 |
| MVP 현실 상한: 폭주H + 검술 + 일격 HH | 10+14 = 24 + 화상 4 = **28** | 폭주H × 일격HH = 1/8 |
| 강타 획득 후: 검술 + 일격 HH + 강타 HH | 10+14+18 = 42 + 화상 3~5 = **45~47** | HH×HH = 1/16 + 화염 배치 |

- **원턴킬 불가**: 최대 분기 47 < 최소 적 HP 60(주술사), 마진 13 (MVP 6종만으로는 최대 38, 마진 22). 적 설계 가드 **일반 적 HP ≥ 55** 유지 (마진 8).
- 폭주 콤보 턴 EV ≈ 24~26 (일반 턴 ~21의 약 1.2배) + 이후 턴 화염 태그/연료 이득 — 일회성(전투당 1회, A17)이므로 "언제 터뜨릴까"가 의사결정이 된다.
- **삼중 안전장치**: ① 턴당 3회(D0) ② 폭주 일회성(A17) ③ 소모 영역 격리(D5). 이를 완화하는 스킬(불의 심장·용의 숨결·손패 생성 희귀)이 추가될 때마다 본 표를 재계산할 것.
- sim 상시 관측: 단일 턴 피해 max가 `0.9 × 최소 적 HP` 초과 시 경보.

## 12. 부록 — 포스트 MVP 예약 지점

MVP에서 구현하지 않되, 데이터 모델이 막지 않도록 자리만 예약한 것들:

- **턴 버프(트리거) 시스템** — 트리거 유형 2종을 예약한다 (효과 원자 `addTurnTrigger`).
  - `onDamageDealt` — **화염검** (PRD v0.4: 화염 코인 **1개** 소비, 고급, "이번 턴 피해를 줄 때마다 화상 +1" — 턴 3회 제한으로 발동 후 공격 기회 최대 2회라 1개가 적정): 자신의 스킬 해결로 적에게 **피해 패킷**이 적용될 때마다 발동(1스킬 = 1패킷 원칙과 정합). 화상 틱·자해·0피해(전부 방어에 흡수돼도 패킷은 성립)는 데이터로 정의.
  - `onAttackSkillResolved` — **불의 심장** ("공격 스킬을 사용할 때마다 화상 +2"): attack 태그 스킬 해결마다 발동, 피해 발생 여부 무관(태그만으로 판정).
  - 공통 규약: 자가 발동 없음(등록 이후 해결되는 스킬부터), 등록 순 발동, 중첩 가산, 턴 종료 소멸.
- **면 조건 확장** — `allHeads/allTails`(§12 확률표의 "모두 앞면 25%" 스킬), 스킬 자체의 속성 조건 라인(§8.1 "속성:").
- **상태이상 일반화** — 컨테이너 2형: 스택형(화상 — 턴 종료 발동 후 −1)과 **지속형**(동상 = n턴 동안 가하는 공격 피해 25% 감소, 감전 = n턴 동안 받는 피해 50% 증가 — PRD v0.3 §6.4). 승산 순서 = 동상(출발) → 감전(수신), floor 반올림, 재부여 시 턴 수 가산. 수정자 훅 + 고정 틱 우선순위 테이블. 화상 틱은 수정자 미적용 고정 피해로 유지(상태이상 간 2차 상호작용 차단).
- **용광로** — `grantElement chooseBasicInHand`(대상 지정형 취급).
- **다중 적 콘텐츠** — 엔진은 이미 지원(A7), 콘텐츠·타겟팅 UI만 추가.
- **다속성 코인** — grants가 집합형이므로 규칙 추가 무비용(C3).
- **콤보 스캐너 / 반사실 리플레이 / CI 밸런스 밴드** — 스킬 10종 이상 시점에 도입. 반사실 리플레이는 플립을 코인 id 키 기반으로 결정하는 구조 변경이 선행 조건.
- **상점·골드·이벤트** (PRD v0.4 §7 스펙 확정, 구현은 Phase 2~3) — 골드 재화(RunState에 `gold` 필드 예약), 상점 3행동(동전 제거 75+25 누진 / 스킬 5진열 / 동전 3진열 — 빌드 추종 가중 롤), 이벤트 위험요소 4유형. 가격·수급 초안은 [content-design-guide.md](./content-design-guide.md) §6. 런 진행을 `combatIndex`가 아닌 노드 리스트로 일반화하는 시점에 함께 구현.
- **조건부 몬스터 행동** (체력 50% 이하/특정 턴 도달 등) + 다중 적 행동 분산 규칙 — 몬스터 양식과 예시 5종(고블린/도적/구울/마도사/슬라임)은 [content-design-guide.md](./content-design-guide.md) §5.
- **영구 변환 보상, 보스 노드** — 상점/이벤트와 같은 노드 일반화 시점에.
